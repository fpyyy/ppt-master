# 大模型 + 奖励机（Large/Foundation Models + Reward Machines）方向调研报告

## 摘要

奖励机（Reward Machine, RM）是一类用有限状态机显式表示非马尔可夫奖励函数的结构化奖励形式。它把长时序任务拆解为若干可解释的自动机状态与转移，从而为强化学习提供子目标、奖励塑形、任务分解和可验证的逻辑结构。近年来，大模型/基础模型（LLM/FM）在自然语言理解、任务分解、代码生成和语义推理方面表现出较强能力，使其成为自动生成奖励机、构造标注函数、设计结构化奖励和进行任务迁移的重要工具。

本调研认为，“大模型 + 奖励机”的核心价值不是简单地让 LLM 直接输出一个奖励函数，而是让 LLM 参与**从自然语言任务到可执行、可检查、可训练的结构化任务规范**的转换。当前研究可以概括为三条主线：

1. **LLM 生成自动机/RM 以辅助强化学习**：例如 LARL-RM 用 LLM 生成 DFA，缩小 RM 学习搜索空间；ARM-FM 进一步从自然语言生成完整的 Language-Aligned Reward Machines。
2. **RM 作为结构化中间表示，支持迁移、泛化与鲁棒学习**：例如 Neural Reward Translation 通过 RM 进行跨域奖励迁移；Noisy RM 工作研究在噪声与部分可观测环境中如何可靠使用 RM。
3. **LLM 自动奖励设计与 RM 方法形成互补**：例如 R* 用 LLM 进行 reward structure evolution 与参数对齐优化，说明 LLM 可以设计高质量 reward，但也暴露了代码式奖励在可验证性、组合性、长期任务结构表达方面的不足。

从研究机会看，该方向尚未完全成熟，尤其在以下问题上仍有空间：LLM 生成 RM 的语义一致性、标注函数 grounding、在线修正与重规划、多智能体任务分配、真实机器人/无人系统中的安全约束建模，以及 RM 与大模型规划能力之间的闭环结合。

## 1. 研究背景

### 1.1 强化学习中的奖励设计问题

强化学习（Reinforcement Learning, RL）通常通过奖励函数引导智能体学习策略。对于简单任务，奖励可以是稀疏的终点奖励，或人工设计的 dense reward。但在长时序任务、组合任务、安全约束任务中，奖励设计会变得困难：

- **稀疏奖励导致探索困难**：智能体可能很长时间都无法获得正反馈。
- **密集奖励容易引入 reward hacking**：手写 shaping reward 可能鼓励错误行为。
- **非马尔可夫任务难以用单步状态奖励表达**：许多任务依赖历史顺序，例如“先侦查，再开门，再救援，再撤离”。
- **真实系统需要可解释、安全、可检查的任务规范**：机器人和无人系统不能只依赖黑箱 reward。

这正是奖励机的切入点：把“任务进度”和“历史依赖”显式编码成有限状态结构，使智能体不仅知道当前状态，还知道任务执行到了哪个阶段。

### 1.2 奖励机的基本思想

奖励机可以理解为一种 Mealy-style automaton，用于表示非马尔可夫奖励函数。一个典型 RM 可写为：

\[
\mathcal{R} = \langle U, u_0, \Sigma, \delta_u, \delta_r \rangle
\]

其中：

- \(U\)：奖励机状态集合；
- \(u_0\)：初始 RM 状态；
- \(\Sigma\)：由命题事件构成的输入字母表，通常来自环境标注函数；
- \(\delta_u\)：RM 状态转移函数；
- \(\delta_r\)：RM 奖励输出函数。

在任务执行过程中，环境状态会被转换为一组高层事件，例如 `has_key`、`door_opened`、`victim_found`、`fire_extinguished`。RM 根据这些事件更新自身状态，并输出相应奖励。

这带来几个优势：

1. **任务结构显式化**：长任务被拆成若干阶段。
2. **奖励更可解释**：每个转移都能对应一个子目标。
3. **可做自动 reward shaping**：中间节点可以给局部奖励。
4. **可做任务分解与技能复用**：不同 RM 片段可以对应不同技能。
5. **适合表达时序逻辑任务**：例如顺序、条件、循环、安全约束等。

### 1.3 为什么需要大模型参与奖励机？

传统 RM 的主要瓶颈在于：RM 结构、事件命题、标注函数和奖励权重往往需要专家手工设计。随着任务变复杂，人工设计成本会迅速上升。

大模型可以在以下环节提供能力：

| 环节 | 大模型的潜在作用 | 风险 |
|---|---|---|
| 自然语言任务理解 | 从任务描述中提取阶段、子目标、约束 | 幻觉、遗漏约束 |
| 自动机/RM 结构生成 | 生成状态、转移、奖励 | 图结构不一致、终止状态错误 |
| 标注函数生成 | 生成 Python/逻辑谓词，用于判断事件是否发生 | grounding 错误、API 不匹配 |
| 任务泛化 | 根据语义相似性复用已有技能 | 语义相似不等于动力学可迁移 |
| 在线修正 | 根据失败轨迹或 counterexample 修改 RM | 修改可能破坏原语义 |

因此，“大模型 + 奖励机”的关键不是让 LLM 完全替代形式化方法，而是把 LLM 作为**任务语义到结构化规范的生成器**，再通过验证、环境反馈、RL 训练和人工检查形成闭环。

## 3. 文献一：ARM-FM —— 基础模型自动生成语言对齐奖励机

### 3.1 基本信息

- **题目**：ARM-FM: Automated Reward Machines via Foundation Models for Compositional Reinforcement Learning
- **作者**：Roger Creus Castanyer, Faisal Mohamed, Pablo Samuel Castro, Cyrus Neary, Glen Berseth
- **出处**：ICLR 2026 Poster
- **关键词**：Foundation Models, Reward Machines, Language-Aligned Reward Machines, Compositional RL, Zero-shot Generalization

### 3.2 研究动机

ARM-FM 的核心问题是：RL 需要高质量奖励规范，但人工设计奖励机成本高；基础模型能够理解自然语言任务，却很难直接转化为可训练的底层奖励信号。因此，该工作试图把基础模型的语义推理能力与 RM 的形式化结构结合起来。

它提出的关键观点是：

> 基础模型适合理解和分解任务；奖励机适合把任务结构转化为可训练、可解释、可复用的 RL 信号。

### 3.3 方法概述

ARM-FM 提出了 **Language-Aligned Reward Machines（LARMs）**。与传统 RM 相比，LARM 不仅包含自动机结构和奖励，还包含自然语言状态描述、可执行标注函数和状态语义嵌入。

一个 LARM 大致包括：

1. **Automaton structure**：状态、初始状态、转移关系。
2. **Reward function**：完成子目标时的中间奖励和最终奖励。
3. **Labeling functions**：由基础模型生成的可执行 Python 谓词，用于检测事件。
4. **Natural-language state instructions**：每个 RM 状态对应一段自然语言说明。
5. **Language embeddings**：把 RM 状态的语言描述编码成向量，作为策略输入的一部分。

训练时，RL agent 不只接收环境 observation，还接收当前 RM 状态的语言嵌入。这样，策略可以根据“当前正在完成哪个子目标”进行条件化决策。

### 3.4 主要贡献

该工作有三个关键贡献：

1. **从自然语言自动生成完整 RM 规范**  
   不只是生成状态转移图，还生成 labeling functions 和状态说明，使 RM 能真正接入 RL 环境。

2. **将 RM 状态语言嵌入加入策略输入**  
   这使得 RM 不再只是静态奖励结构，而是可以成为“技能语义条件”。策略可以学习“在某类子目标语义下应如何行动”。

3. **展示组合泛化与 zero-shot generalization**  
   如果新任务由训练时见过的子任务组合而成，agent 可以在没有额外微调的情况下利用新的 RM 完成新组合任务。

### 3.5 实验与结果解读

ARM-FM 在 MiniGrid、Craftium/Minecraft-like resource gathering 等长时序稀疏奖励任务上验证方法有效性。论文强调：在复杂稀疏奖励任务中，普通 RL agent 难以取得进展，而 FM-generated LARM 能提供中间奖励和任务结构，使 agent 更快学习。

尤其值得关注的是其 zero-shot generalization 实验：agent 在若干任务上训练后，给定一个新的组合任务 RM，可以利用语义相近的已学技能完成新任务。这对多阶段机器人任务非常有吸引力，因为现实任务往往不是完全新任务，而是已有技能的重组。

### 3.6 局限性

ARM-FM 虽然是当前最完整的“大模型+RM”框架，但仍有明显局限：

1. **仍需要验证环节**  
   论文承认 human verification 可以提高生成质量。也就是说，LLM 生成 RM 并不是天然可靠的。

2. **标注函数 grounding 是瓶颈**  
   在仿真环境中，Python API 明确，LLM 可以生成谓词；但真实机器人环境中，`victim_found`、`door_locked`、`fire_blocked` 等事件需要由视觉、语音、传感器或多模态模型判断，误差更大。

3. **奖励数值仍有人为痕迹**  
   中间奖励权重如何设定仍可能影响策略学习，且可能引入局部最优。

4. **在线动态任务变化讨论不足**  
   该框架更偏向离线生成任务 RM，然后训练 agent。对于任务执行中实时发现新信息并重构 RM 的场景，仍需要进一步扩展。

### 3.7 对本方向的启发

ARM-FM 可以作为“大模型+奖励机”方向的主线文献。它证明了一个重要趋势：

> 未来的任务规范可能不是人工手写 RM，而是由大模型从自然语言生成 RM，再通过形式化检查、环境反馈和人工校验逐步修正。

对于多机器人救援、无人系统任务规划等场景，可以考虑用类似 LARM 的结构表达“侦查—破门—搜索—灭火—救援—撤离”等任务链，并让不同 RM 状态对应不同机器人技能或子策略。

## 4. 文献二：LARL-RM —— LLM 生成 DFA 辅助奖励机学习

### 4.1 基本信息

- **题目**：Using Large Language Models to Automate and Expedite Reinforcement Learning with Reward Machine
- **作者**：Shayan Meshkat Alsadat, Jean-Raphaël Gaglione, Daniel Neider, Ufuk Topcu, Zhe Xu
- **出处**：ACC 2025；arXiv 2024
- **关键词**：LLM-generated DFA, Reward Machine Learning, Prompt Engineering, Counterexample, Closed-loop RL

### 4.2 研究动机

该论文关注的问题是：已有 RM 学习方法可以从交互轨迹中学习奖励机，但通常需要专家提供 advice 或高层结构约束。作者希望用 LLM 替代专家，自动生成 high-level automaton knowledge，从而加速 RM 学习和 RL 训练。

与 ARM-FM 直接生成完整 RM 不同，LARL-RM 更像是：

> LLM 先生成一个 DFA 作为高层先验，用来约束或指导 RM 学习；如果该 DFA 与实际轨迹不兼容，则通过 counterexample 更新 prompt 或丢弃错误 DFA。

### 4.3 方法概述

LARL-RM 的核心流程如下：

1. 给 LLM 输入自然语言任务描述和相关示例。
2. LLM 生成一个 DFA，表示哪些事件序列可能通向正奖励。
3. RL/RM learning algorithm 使用该 DFA 缩小搜索空间。
4. 如果生成 DFA 与观察到的样本轨迹不兼容，则使用 counterexample 反馈给 LLM 或移除该 DFA。
5. 通过闭环过程逐步获得更有用的自动机先验。

这个流程的关键不是“相信 LLM”，而是“把 LLM 输出当作可验证的候选约束”。这点很重要，因为它降低了 LLM 幻觉的风险。

### 4.4 主要贡献

1. **把 LLM 输出转化为 DFA 形式的 high-level advice**  
   这使得自然语言知识可以进入 RM learning pipeline。

2. **提出 open-loop 与 closed-loop 两种配置**  
   open-loop 只生成一次 DFA；closed-loop 可以根据 counterexample 更新 prompt。

3. **与已有 RM learning 方法结合**  
   LLM 不直接替代学习算法，而是提供搜索偏置或结构先验。

4. **理论上讨论收敛性**  
   论文给出条件下的收敛保证，强调不兼容 DFA 会被检测并移除。

### 4.5 实验与结果解读

论文通过两个 case studies 展示：当 LLM-generated DFA 与真实 RM 兼容时，学习可以更快收敛；当 DFA 初始不兼容时，closed-loop 机制可以利用 counterexample 调整 prompt，生成更合适的 DFA。

论文报告 LARL-RM 可以加快收敛，并展示了 open-loop/closed-loop 对比。虽然实验规模较小，但它明确提出了一个重要范式：

> LLM 不是直接输出最终答案，而是输出可被 RM learner 验证、利用、修正的结构化先验。

### 4.6 局限性

1. **实验环境较简单**  
   论文主要是概念验证，还没有覆盖复杂机器人、多智能体、大规模任务图。

2. **DFA 与 RM 的表达关系需要谨慎处理**  
   DFA 主要表达接受语言，RM 还需要奖励输出；二者不是完全等价。

3. **命题集合和环境标签仍依赖人工设计**  
   如果 proposition space 不合理，LLM 生成 DFA 的意义有限。

4. **LLM prompt 稳定性不足**  
   不同 prompt、模型版本、采样温度可能生成不同结构，工程复现需要额外控制。

### 4.7 对本方向的启发

LARL-RM 给出了一个比“直接让 LLM 生成 RM”更稳健的思路：LLM 生成候选结构，算法负责验证和学习。这对于安全关键任务尤其重要。

在实际研究中，可以把该思路扩展为：

```text
任务描述 / 成功轨迹 / 失败轨迹
        ↓
LLM 生成候选 DAG / DFA / RM
        ↓
形式化检查 + 轨迹一致性检查
        ↓
不一致则反馈 counterexample
        ↓
生成修正后的 RM
        ↓
RL 或多智能体执行
```

## 5. 文献三：Reward Translation via Reward Machine —— RM 作为跨域奖励迁移桥梁

### 5.1 基本信息

- **题目**：Reward Translation via Reward Machine in Semi-Alignable MDPs
- **作者**：Yun Hua, Haosheng Chen, Wenhao Li, Bo Jin, Baoxiang Wang, Hongyuan Zha, Xiangfeng Wang
- **出处**：ICML 2025
- **关键词**：Reward Translation, Semi-alignable MDPs, Reward Machines, Cross-domain Transfer, LLM-based Automated Learning

### 5.2 研究动机

该论文研究的是跨域奖励迁移。传统奖励迁移通常要求源域和目标域之间有较强对齐关系，例如状态之间可建立双射，或者有时间对齐的 demonstration。但现实中，不同环境可能只有粗粒度结构相似：任务阶段类似，但具体状态空间、动作空间、动力学不同。

因此，作者提出 semi-alignable MDPs，并用 RM 作为跨域中间表示：

> 直接在状态空间中迁移奖励很难；但可以先把任务抽象成 RM，再在 RM 图结构之间做匹配，从而实现奖励迁移。

### 5.3 方法概述

该工作提出 Neural Reward Translation（NRT）框架。核心步骤是：

1. 在源域和目标域中构造或学习对应 RM。
2. 通过图匹配方法建立两个 RM 之间的结构对应关系。
3. 借助 RM 对齐结果，把源域奖励翻译到目标域。
4. 在目标域中用翻译后的奖励加速 RL 学习。

论文中特别指出，RM 可以通过有限人工输入或 LLM-based automated learning 构造。这使它与“大模型+RM”方向产生关联：LLM 可用于把自然语言或轨迹转换为 RM，而 RM 再作为跨域迁移的结构桥梁。

### 5.4 主要贡献

1. **定义 reward translation 问题**  
   把跨域奖励迁移形式化为 semi-alignable MDPs 下的奖励翻译。

2. **使用 RM 进行间接映射**  
   不要求源域和目标域状态一一对应，而是通过 RM 抽象任务结构。

3. **结合图匹配实现 RM 对齐**  
   任务结构相似即可迁移，而不要求底层状态空间一致。

4. **扩展 RM 的应用场景**  
   RM 不只是单任务奖励规范，还可以成为跨域知识迁移载体。

### 5.5 局限性

1. **依赖 RM 构造质量**  
   如果源域或目标域 RM 构造错误，后续图匹配会被放大错误。

2. **图匹配不等于语义匹配**  
   图结构相似不一定代表子任务语义一致。

3. **LLM 自动构造 RM 只是辅助设定，不是核心验证对象**  
   论文重点是 reward translation，而不是系统评估 LLM 生成 RM 的可靠性。

4. **对于多智能体任务迁移仍需扩展**  
   不同 agent 的能力差异、协作关系和资源约束尚未充分建模。

### 5.6 对本方向的启发

这篇论文的重要性在于提出了 RM 的另一个角色：

> RM 不仅是 reward function，也是跨任务、跨环境、跨机器人平台的结构化任务抽象。

例如在救援任务中，无人机、无人车、机器狗可能拥有不同动作空间，但高层任务结构可能相似：侦查、定位、清障、接近、救援、撤离。RM 可以作为这些平台之间共享的任务语义骨架，而具体执行策略由不同 agent 根据自身能力学习或调用。

## 6. 文献四：Reward Machines for Deep RL in Noisy and Uncertain Environments —— 现实环境中的 RM grounding 问题

### 6.1 基本信息

- **题目**：Reward Machines for Deep RL in Noisy and Uncertain Environments
- **作者**：Andrew C. Li, Zizhao Chen, Toryn Q. Klassen, Pashootan Vaezipoor, Rodrigo Toro Icarte, Sheila A. McIlraith
- **出处**：NeurIPS 2024 Poster
- **关键词**：Reward Machines, Noisy Labels, Partial Observability, POMDP, Formal Language, Automata

### 6.2 为什么这篇非 LLM 论文也必须纳入？

表面上看，这篇论文不是“大模型+RM”论文；但它解决的是“大模型+RM”落地中绕不开的问题：**RM 的命题事件从哪里来，以及它们是否可靠？**

无论 RM 是人工写的，还是 LLM 生成的，都需要一个 labeling function 把环境状态映射到命题事件。例如：

```text
image / sensor / state  →  {door_open, victim_detected, fire_blocked}
```

在真实环境中，这个过程可能由视觉模型、传感器融合、多模态大模型或规则系统完成。问题是：这些判断通常有噪声，且在部分可观测环境中不确定。

因此，该论文研究的是 RM 在 noisy and uncertain grounding 下如何与 deep RL 结合。

### 6.3 方法概述

该工作把带噪声命题解释的 RM 环境形式化为 POMDP，并提出一组利用 RM 结构的 deep RL 算法。核心思想是：

1. RM 的高层结构仍然有价值；
2. 但不能假设命题标签完全正确；
3. 应把命题解释的不确定性显式纳入学习过程；
4. naive 地反复查询同一个 noisy abstraction model 可能带来相关误差，而不是独立误差。

这对基于 LLM/VLM 的 RM 系统特别重要。因为 LLM 或视觉模型输出的事件判断往往不是严格真值，而是概率性、上下文依赖或有偏的估计。

### 6.4 主要贡献

1. **提出 noisy RM setting**  
   将 RM 从理想的完全可观测命题解释扩展到带噪声、部分可观测环境。

2. **把问题刻画为 POMDP**  
   为后续算法设计提供形式化基础。

3. **分析 naive 方法的风险**  
   如果错误地把重复查询结果当作 i.i.d.，可能导致严重偏差甚至危险行为。

4. **实验验证 RM 结构在不确定标签下仍能提高学习效率**  
   即使命题解释不完美，任务结构仍然可以被利用。

### 6.5 局限性

1. **没有直接处理 LLM 生成 RM 的结构错误**  
   论文主要处理命题解释不确定，而不是 RM 图结构错误。

2. **未充分考虑在线 RM 更新**  
   如果 RM 本身需要动态修改，该框架还需扩展。

3. **模型不确定性与语义不确定性仍可进一步区分**  
   视觉误检、语言歧义、任务规范歧义是不同层面的不确定性。

### 6.6 对本方向的启发

如果要把“大模型+奖励机”用于真实机器人或无人系统，这篇论文提供了必要警示：

> 不能只研究 LLM 能否生成好看的 RM 图；还必须研究 RM 中每个命题是否能被可靠检测。

对于真实场景，建议把 RM 系统拆成三层：

```text
自然语言任务层：任务目标、约束、偏好
        ↓
RM 结构层：状态、转移、奖励、终止条件
        ↓
命题 grounding 层：视觉/传感器/多模态模型判断事件
```

其中第三层是最容易被忽视、但最影响实际效果的部分。

## 7. 文献五：R* —— LLM 自动奖励设计作为对照路线

### 7.1 基本信息

- **题目**：R*: Efficient Reward Design via Reward Structure Evolution and Parameter Alignment Optimization with Large Language Models
- **作者**：Pengyi Li, Jianye Hao, Hongyao Tang, Yifu Yuan, Jinbin Qiao, Zibin Dong, Yan Zheng
- **出处**：ICML 2025
- **关键词**：LLM-based Reward Design, Reward Structure Evolution, Parameter Alignment, Preference Learning, Robotic Control

### 7.2 研究动机

R* 不使用奖励机作为核心表示，而是研究 LLM 如何自动设计高质量 reward functions。它的动机是：LLM 具有代码生成能力和领域知识，可以自动生成奖励函数；但代码式奖励设计空间巨大，既包括结构设计，也包括参数配置，直接让 LLM 一次性生成 reward 往往效率低、不稳定。

### 7.3 方法概述

R* 将自动奖励设计拆成两个子问题：

1. **Reward Structure Evolution**  
   维护一个 reward function population，用 LLM 作为 mutation operator，并通过模块级 crossover 探索不同 reward 结构。

2. **Parameter Alignment Optimization**  
   LLM 生成多个 critic functions，对轨迹片段进行比较和标注；通过投票获得高置信偏好标签，再用 preference learning 调整 reward 参数。

简单来说，R* 不是让 LLM 直接给出一个最终 reward，而是让 LLM 参与 reward 结构搜索和参数对齐。

### 7.4 主要贡献

1. **把 reward design 拆成结构和参数两个层次**  
   这比一次性生成 reward 更稳定。

2. **使用 LLM 作为 reward mutation operator**  
   体现出 LLM 在代码式 reward search 中的作用。

3. **用偏好学习对齐 reward 参数**  
   减少手动调参依赖。

4. **在机器人控制任务中超过强基线和人工设计 reward**  
   说明 LLM-based reward design 已具备实际竞争力。

### 7.5 与 RM 路线的关系

R* 与 RM 路线形成明显对比：

| 维度 | R* 代码式 reward | RM 结构化 reward |
|---|---|---|
| 表示形式 | Python/函数式 reward | 有限状态机 + 命题逻辑 |
| 长时序结构 | 隐含在 reward 代码中 | 显式在状态转移中 |
| 可解释性 | 取决于代码质量 | 通常更强 |
| 形式化验证 | 较难 | 相对更自然 |
| 组合泛化 | 需要额外机制 | RM 天然支持子任务组合 |
| 工程灵活性 | 强 | 中等，依赖命题设计 |
| 适用任务 | 连续控制、机器人技能 | 长时序、逻辑约束、任务规划 |

因此，R* 的意义不是替代 RM，而是说明另一条路径：LLM 可以优化 reward structure。未来可以考虑把二者结合：

```text
LLM 生成 RM 骨架
        ↓
RM 定义任务阶段与逻辑约束
        ↓
LLM/R* 风格方法为每个 RM 转移或子任务设计局部 reward
        ↓
偏好学习或轨迹反馈优化 reward 权重
```

这可能比单独使用 RM 或单独使用代码 reward 更稳健。

## 8. 横向对比

### 8.1 方法对比表

| 文献 | LLM/FM 的角色 | RM 的角色 | 是否直接生成 RM | 是否处理不确定性 | 是否强调迁移/泛化 | 主要适用场景 |
|---|---|---|---|---|---|---|
| ARM-FM | 从自然语言生成 LARM、labeling functions、状态说明 | 核心任务规范与策略条件 | 是 | 部分考虑验证，但不主攻噪声 | 强，zero-shot compositional generalization | 长时序稀疏奖励、组合任务 |
| LARL-RM | 生成 DFA 作为 high-level advice，并可用 counterexample 修正 | 被学习对象/结构化奖励 | 间接生成 DFA，辅助 RM learning | 处理 DFA 不兼容，但非感知噪声 | 弱 | RM 学习加速 |
| NRT | 可用于自动构造 RM | 跨域奖励迁移桥梁 | 可选，非核心 | 不主攻 | 强，跨域迁移 | semi-alignable MDPs |
| Noisy RM | 无 LLM | 在噪声命题解释下仍作为任务结构 | 否 | 强 | 中等 | 真实传感器/视觉噪声环境 |
| R* | 生成/变异 reward 代码，生成 critic functions | 无 RM | 否 | 通过偏好标签提高鲁棒性 | 中等 | 机器人连续控制、reward design |

### 8.2 技术路线演化

可以把该方向的发展理解为以下脉络：

```text
人工设计奖励函数
        ↓
结构化奖励：Reward Machine
        ↓
自动学习/推断 Reward Machine
        ↓
LLM 生成 DFA/RM 先验：LARL-RM
        ↓
Foundation Model 生成完整 Language-Aligned RM：ARM-FM
        ↓
面向真实系统：处理 noisy grounding、在线修正、多智能体协作
```

同时，LLM 自动 reward design 路线也在并行发展：

```text
人工 dense reward
        ↓
LLM 生成 reward code
        ↓
reward structure evolution + preference alignment
        ↓
与 RM 结合：RM 负责时序结构，LLM reward 负责局部连续控制细节
```

## 9. 研究趋势总结

### 9.1 趋势一：从“语言到计划”走向“语言到可训练奖励结构”

早期 LLM+RL 工作更多关注让 LLM 输出计划、子目标或动作建议。但计划本身不一定能训练 RL agent。RM 提供了一个更适合学习的中间层：既保留任务结构，又能输出奖励信号。

ARM-FM 的意义就在于把自然语言任务转化为可执行的 RM，包括结构、标注函数和状态语义，使语言理解真正进入 RL training loop。

### 9.2 趋势二：LLM 输出必须进入验证闭环

LARL-RM 和 ARM-FM 都体现了一个共识：不能无条件相信 LLM 输出。

有效做法包括：

- 用 counterexample 反馈修正；
- 用轨迹一致性检查筛掉错误自动机；
- 用人工或程序检查 labeling functions；
- 用形式化约束检查 RM 是否满足基本性质；
- 对 RM 进行最小化、可达性检查和死状态检查。

因此，该方向更合理的范式是：

```text
LLM proposes → verifier checks → environment tests → learner adapts → LLM revises
```

而不是：

```text
LLM directly outputs final RM
```

### 9.3 趋势三：grounding 将成为核心难点

在仿真环境中，命题事件通常很容易判断。但在真实机器人任务中，命题 grounding 是最大问题之一。

例如：

- `victim_detected` 可能依赖视觉检测；
- `room_safe` 可能依赖烟雾、温度、结构稳定性估计；
- `door_locked` 可能依赖触觉、视觉和动作反馈；
- `path_blocked` 可能随时间变化；
- `fire_extinguished` 可能不是二值事件，而是概率状态。

Noisy RM 文献说明：如果命题解释有噪声，RM 结构仍可利用，但算法必须显式建模不确定性。

### 9.4 趋势四：RM 适合作为多任务、多域、多智能体的共享抽象

NRT 说明 RM 不只是 reward，还可以作为跨域迁移中间表示。对于多智能体系统，RM 也可以作为共享任务图，然后根据不同 agent 能力进行分配。

例如一个救援任务 RM 可以包含：

```text
u0: 初始
u1: 完成区域侦查
u2: 发现受困者
u3: 判断伤情
u4: 轻伤救助并引导撤离
u5: 重伤转运
u6: 火源隔离/灭火
u7: 全员撤离完成
```

无人机、无人车、机器狗可以共享同一 RM，但执行不同边上的子任务。

### 9.5 趋势五：RM 与代码式 reward design 可能融合

R* 说明 LLM 自动 reward design 已经能在机器人控制任务中达到较强效果。但代码式 reward 在长时序逻辑任务上缺乏 RM 的显式结构；RM 在连续控制和局部技能细节上又可能过于抽象。

未来更有价值的方向可能是：

- RM 负责高层任务结构；
- LLM 生成或优化每条 RM 边对应的局部 reward；
- 轨迹偏好学习调整 reward 参数；
- 多模态模型负责事件检测；
- verifier 保证 RM 结构不违反约束。

## 10. 关键开放问题

### 10.1 LLM 生成 RM 的语义一致性

LLM 生成的 RM 可能出现：

- 多个不合理终止状态；
- 死状态或不可达状态；
- 转移条件重叠或缺失；
- 子任务顺序与自然语言不一致；
- 安全约束被遗漏；
- 图结构过大、不可解释。

因此需要 RM-level validation，包括：

1. 可达性检查；
2. 终止性检查；
3. 转移完备性检查；
4. 转移互斥性检查；
5. 与示例轨迹的一致性检查；
6. 与 LTL/自然语言约束的一致性检查；
7. 奖励单调性或阶段进展检查。

### 10.2 标注函数 grounding

LLM 可以生成 `def has_key(env)` 这种函数，但真实环境中没有干净的 `env` API。需要研究：

- 如何从视觉/多模态模型获得命题概率；
- 如何把概率命题接入 RM；
- 如何处理命题延迟、漏检、误检；
- 如何在任务执行中更新命题置信度；
- 如何避免 noisy predicate 导致 RM 状态错误跳转。

### 10.3 在线 RM 修正与任务重规划

真实任务不是静态的。例如救援场景中，无人机可能在执行中发现新火源、新障碍、额外受困者。此时 RM 需要在线更新。

关键问题：

- 修改 RM 时如何保持已完成任务状态？
- 新 RM 如何继承旧 RM 的进度？
- 如何避免频繁重规划导致策略不稳定？
- 修改局部边是否会破坏全局语义？
- 多智能体如何同步 RM 状态？

### 10.4 多智能体 RM 分解

对于 UAV/UGV/机器狗协作任务，单一 RM 只能描述全局任务进展，但执行需要分配到不同 agent。

可以研究：

- 全局 RM → agent-specific RM；
- RM 边 → agent capability matching；
- 并行子任务的 RM 表达；
- 通信失败时的 RM 状态同步；
- 安全约束 RM 与任务 RM 的组合；
- 任务 RM（TRM）与结构/安全 RM（SRM）的分离。

### 10.5 LLM 规划与 RM 学习之间的分工

一个合理分工可能是：

| 模块 | 功能 |
|---|---|
| LLM/FM | 从任务描述、现场信息、历史轨迹中生成候选任务结构 |
| RM verifier | 检查图结构、逻辑约束、轨迹一致性 |
| Grounding module | 把传感器输入转化为命题事件或概率 |
| RL module | 学习子策略或执行策略 |
| Online replanner | 根据新观测修正 RM |
| Human operator | 审核高风险任务约束 |

这种结构比单独依赖 LLM 或单独依赖 RL 更适合安全关键任务。
